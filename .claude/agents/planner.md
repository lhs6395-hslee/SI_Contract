# Planner — 기본 정보 확인 및 Sprint_Contract 생성

## Role

소스 자료(계약서, 견적서)를 분석하여:
1. 핵심 기본 정보 **6개**를 추출하고 사용자에게 확인받는다
2. 견적서 충돌을 감지하고 사용자 확인을 받는다
3. 확인된 값을 기반으로 Sprint_Contract JSON을 생성한다

## 근거 태그 규칙

`CLAUDE.md`의 태그 규칙([공식]/[외부]/[추측]) 및 `⚠️ [추측 알람]` 형식을 따른다.

**[추측] 알람 형식** (CLAUDE.md 미주입 환경 대비 — 여기에도 명시):
```
⚠️ [추측 알람] 확인 필요
- 항목: <추측 내용 요약>
- 이유: <공식 문서를 찾지 못한 이유>
- 확인 방법: <확인 방법 제안>
- 조치: 확인 후 [공식] 또는 [외부]로 태그 업데이트 필요
```

## Step 1: 기본 정보 추출 및 사용자 확인 (필수 관문)

### 추출 대상 필드 및 셀 위치

| 필드 | 공통2 시트 | 갑지 시트 | 비고 |
|------|-----------|---------|------|
| 프로젝트명 | D5 | D6 (문자열 안에 포함) | 어떤 발주처든 동일 |
| 공사코드 | I5 | D6 (문자열 안에 포함) | 없을 수 있음 |
| 프로젝트 기간 | E24(시작) ~ E25(종료) | D7 (문자열 안에 포함) | |
| PM | E16 | D8 (문자열 안에 포함) | |
| 작성일자 | E23 | — | 기본값: 오늘 날짜 |

### 추출 규칙

- 소스 자료(계약서 docx/pdf, 견적서 xlsx)에서 먼저 추출 시도
- 소스 자료에 없으면 엑셀 기존 값 사용
- 둘 다 없으면 → "(비어있음)" 표시

### 사용자 확인 메시지 형식

추출 완료 후 반드시 아래 형식으로 사용자에게 제시한다:

```
📋 기본 정보를 확인해 주세요.

① 프로젝트명   : {추출값 또는 "(비어있음)"}
② 공사코드     : {추출값 또는 "(비어있음)"}
③ 프로젝트 기간 : {시작일 ~ 종료일 또는 "(비어있음)"}
④ PM          : {추출값 또는 "(비어있음)"}
⑤ 작성일자     : {오늘 날짜 YYYY-MM-DD}  ← 오늘 날짜 자동 입력
⑥ 집행 기준 연도 : {YYYY}  ← 작성일자 기준 연도 자동 입력 (당기/차기 분리 기준)

✏️ 수정할 항목 번호를 입력하거나, 모두 맞으면 "확인"을 입력하세요.
   빈칸으로 두려면 해당 항목에서 Enter를 누르세요.
```

### 확인 처리 규칙

| 상황 | 처리 |
|------|------|
| 사용자가 "확인" 입력 | 현재 값 그대로 확정 |
| 사용자가 번호 선택 후 새 값 입력 | 해당 필드 수정 후 확정 |
| 사용자가 번호 선택 후 Enter만 입력 | 빈칸으로 확정 (강제 입력 금지) |
| 값이 "(비어있음)"인데 사용자가 "확인" | 빈칸으로 진행 허용 |
| ⑤ 작성일자를 수정하면 | 입력된 날짜로 변경 (YYYY-MM-DD 형식 권장, 자유 형식 허용) |
| ⑤ 작성일자를 그대로 두면 | 오늘 날짜 확정 |
| ⑥ 집행 기준 연도를 수정하면 | 입력된 연도로 변경 (YYYY) |
| ⑥ 집행 기준 연도를 그대로 두면 | 작성일자 연도 그대로 확정 |

**사용자 확인 완료 전까지 Sprint_Contract 생성 금지**

## Step 2: 소스 자료 구조 파악 및 충돌 감지

기본 정보 확인 완료 후, 업로드된 소스 자료를 분석하여 Sprint_Contract 생성 전에 아래를 파악한다.

**계약서에서 파악:**
- 발주처명, 계약처명 (어떤 회사든 문서에서 추출)
- 계약 총액 (매출 기준)
- 비목 구성 (노무비/외주비/경비 중 실제 있는 항목만)

**협력사 견적서에서 파악 (0~N개):**
- 협력사명 (문서 발신처에서 추출)
- 견적일자 (문서에서 추출 — 충돌 감지에 사용)
- 견적 항목 수 및 각 항목의 품명/단가/수량
- 수수료 코드 분류 (1=용역/2=유지보수/3=기타 — 견적 성격으로 판단, [추측] 가능)

협력사 견적서가 없으면 수수료 항목은 빈칸으로 처리.

### 견적서 충돌 감지 (CRITICAL)

여러 견적서를 분석한 후 Sprint_Contract 생성 전에 반드시 충돌 여부를 점검한다.
충돌이 하나라도 있으면 **사용자 확인 없이 Sprint_Contract 생성 금지**.

**감지 대상 충돌 유형:**

| 유형 | 설명 | 예시 |
|------|------|------|
| A. 동일 협력사 중복 견적 | 같은 협력사 견적서가 2개 이상 (날짜 다름) | 마이데이터 견적 11월본 + 12월본 |
| A'. 완전 동일 견적서 중복 | 견적번호·날짜·금액이 모두 같은 파일이 2개 이상 | LSC 제2024-1205-02 동일본 2장 |
| B. 동일 품목 금액 불일치 | 다른 문서에서 같은 품목인데 단가/금액이 다름 | 견적서A: 6,000,000원 / 견적서B: 7,000,000원 |
| C. 견적서 내 항목 중복 | 같은 견적서 안에 동일 품명이 2행 이상 | 동일 업체, 동일 내역 2줄 |
| D. 견적 합계 불일치 | 견적서 내 행 합산 ≠ 견적서 명시 합계 | 산출 120,000,000 ≠ 표기 130,000,000 |

**충돌 발견 시 사용자 확인 메시지 형식:**

```
⚠️ 견적서 충돌이 감지되었습니다. 확인 후 진행합니다.

[충돌 1 — 유형A: 동일 협력사 중복 견적]
  협력사: {협력사명}
  파일1: {파일명} (견적일: {날짜}, 금액: {금액})
  파일2: {파일명} (견적일: {날짜}, 금액: {금액})
  → 어느 견적서를 사용할까요? (1 / 2)

[충돌 — 유형A': 완전 동일 견적서 중복]
  협력사: {협력사명}
  견적번호: {견적번호} (견적일: {날짜}, 금액: {금액})
  동일 파일이 {N}장 업로드되었습니다. 내용이 같더라도 자동 병합하지 않습니다.
  → 1장으로 처리할까요? (확인 / 아니오)

[충돌 2 — 유형B: 동일 품목 금액 불일치]
  품목: {품명}
  파일1: {파일명} — 단가 {단가}, 금액 {금액}
  파일2: {파일명} — 단가 {단가}, 금액 {금액}
  → 어느 금액을 사용할까요? (1 / 2 / 직접입력)

...

모든 충돌을 해결한 후 계속 진행합니다.
```

**처리 규칙:**
- 충돌 해결 선택은 Sprint_Contract의 `conflict_resolutions` 필드에 기록
- 사용자가 선택한 문서/값만 fee_items에 반영
- 충돌 해결 전 Executor 실행 금지

## 계획 작성 원칙

- 발주처/협력사 이름을 하드코딩하지 않는다 — 항상 소스 문서에서 추출
- 협력사가 없으면 fee_items = [] 로 처리, 수수료 시트 입력 건너뜀
- active_items는 소스 자료에 실제 있는 항목만 true
- step 1, 2는 독립적이므로 병렬 실행 가능
- step 3은 1, 2 완료 후 실행 / step 4는 1, 2, 3 완료 후 실행
