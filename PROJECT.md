# SI Contract — 프로젝트 요구사항 문서

## 프로젝트 개요

SI(System Integration) 집행계획서 자동 작성 파이프라인.

**문제**: 프로젝트마다 계약서/견적서 형식이 달라 집행계획서를 매번 수기로 작성해야 하는 번거로움.
**해결**: 소스 자료(계약서, 견적서)를 업로드하면 AI가 문맥을 해석하여 고정 양식인 집행계획서 엑셀의
정해진 필드에 값을 추출·매핑하고, 사용자 확인 후 최종 입력한다.

**범용 설계 원칙**:
- 발주처/계약처는 어떤 회사든 대응 (삼성전자, LG전자, SK 등 고정값 없음)
- 협력사(수수료 매입처)는 0개~N개 대응 (없을 수도, 여러 개일 수도 있음)
- 수수료 항목 수는 가변 (1줄~N줄)
- 비목 구성은 프로젝트마다 다를 수 있음 (노무비만 있을 수도, 외주비가 있을 수도)

기술 스펙 레퍼런스: `specs/ai-agent-engineering-spec-2026.md`

## 작업 흐름 (3단계 사용자 확인)

```
1. 사용자가 소스 자료 업로드
   - 계약서 (docx / pdf) — 발주처와 우리 회사 간 계약
   - 협력사 견적서 (xlsx / pdf) — 0개~N개, 없을 수 있음

2. Planner → 기본 정보 추출 및 사용자 확인 (필수 관문)
   - 소스 자료에서 6개 핵심 필드 추출:
     · 프로젝트명   (공통2 D5)
     · 공사코드     (공통2 I5)
     · 프로젝트 기간 (공통2 E24~E25)
     · PM          (공통2 E16)
     · 작성일자     (공통2 E23) ← 기본값: 오늘 날짜
     · 집행 기준 연도 ← 작성일자 연도 자동 입력 (당기/차기 이월 분리 기준)
   - 추출된 값이 있으면: 보여주고 확인 요청
   - 추출된 값이 없으면: "(비어있음)" 표시 후 입력 요청
   - 빈칸 허용: 사용자가 공란으로 두겠다고 하면 그대로 진행
   - 사용자 확인 완료 전까지 다음 단계 진행 금지

3. Planner → 견적서 충돌 감지 및 사용자 확인 (필수 관문)
   - 여러 견적서 간 충돌 유형 자동 감지:
     · 유형A: 동일 협력사 견적서가 2개 이상 (날짜 다름)
     · 유형A': 완전 동일 견적서가 2장 이상 (내용 같아도 자동 병합 금지)
     · 유형B: 다른 문서에 같은 품목인데 금액이 다름
     · 유형C: 같은 견적서 안에 동일 품명 중복
     · 유형D: 견적서 행 합산 ≠ 견적서 명시 합계
   - 충돌 발견 시 → 사용자에게 어느 값을 쓸지 확인 (자동 판단 금지)
   - 충돌 없으면 → 바로 Sprint_Contract 생성
   - 사용자 해결 완료 전 Sprint_Contract 생성 금지

4. Planner → Sprint_Contract 생성
   - 확인된 기본 정보 + 충돌 해결 결과를 고정값으로 기록
   - 소스 자료에서 파악된 비목 구성 반영 (있는 비목만 처리)

5. Executor → 소스 자료 분석 및 값 추출
   - 각 필드별 추출값 + 근거(어느 문서 어느 부분) 제시
   - 불확실한 항목은 [추측] 알람 포함

6. 사용자 확인
   - 추출 결과 전체 검토
   - [추측] 항목 직접 확인 및 수정

7. 사용자 승인 후 → Executor가 집행계획서 엑셀에 값 입력

8. Reviewer → 5단계 금액 검증 (독립 실행, 상세: reviewer.md 참조)

   8-1. 수수료 구조 검증: 행별 수량×단가=금액, 소계/합계, 매출≥매입
   8-2. 충돌 해결 검증: 사용자 선택값이 실제 입력값과 일치하는지 확인
   8-3. 산출내역서 검증: 노무비/경비 합산, 수수료↔5.4 교차
   8-4. 갑지 검증: 비목별 합산, 매출액=계약서 총액, 영업이익 계산
   8-5. 기본 정보 검증: 사용자 확인값과 입력값 일치

   ※ 수수료 매출단가 ≠ 매입단가는 정상 (마진 구조)
   ※ 매입단가 > 매출단가는 역마진 → FAIL

9. 검증 실패 시 → Executor 재수정 (최대 3회)
   동일 오류 2회 연속 → 사용자 에스컬레이션

10. 최종 결과물 반환
```

### 5.4 수수료 (가변 구조 — N행)

수수료 항목 수는 프로젝트마다 다름. 협력사 견적서 수에 따라 행 수 결정.

| 필드 | 소스 | 비고 |
|------|------|------|
| 수수료 코드 | 견적서 성격 판단 | 1=용역, 2=유지보수, 3=기타 |
| 품명 | 협력사 견적서 | 견적서 상세내역명 |
| 규격 | 협력사 견적서 | |
| 단위 | 협력사 견적서 | M/M, 식, 건 등 |
| 계약 수량/단가/금액 | **발주처 계약서** | 매출 기준 |
| 집행 수량/단가/금액 | **협력사 견적서** | 매입 기준 |
| 당기계획 수량/금액 | 협력사 견적서 | 해당 연도 집행분 |

**핵심**: 계약(매출) ≠ 집행(매입) — 차이가 마진. 역전 시 오류.

### 5. 산출내역서 (있는 비목만 입력)

| 비목 | 입력 조건 | 소스 |
|------|---------|------|
| 재료비 | 재료비 항목이 있을 때만 | 견적서 |
| 노무비 - 급료 | 직원 투입 시 | 계약서/사내 기준 |
| 노무비 - 상여금 | 해당 시 | 사내 기준 |
| 노무비 - 퇴직금 | 해당 시 | 급여/12 계산 |
| 외주비 | 외주 항목 있을 때만 | 견적서 |
| 경비 - 복리후생비 | 해당 시 | 인원×단가 계산 |
| 경비 - 보험료 | 노무비 있을 때 | 요율 계산 |
| 경비 - 수수료 | 협력사 있을 때 | 5.4 수수료 합계 |
| 경비 - 기타 | 해당 항목별 | 각 소스 |

### 0. 집행(갑지)

| 필드 | 셀 | 소스 |
|------|-----|------|
| 프로젝트명+공사코드 | D6 | 공통2 참조 |
| 프로젝트 기간 | D7 | 공통2 참조 |
| PM | D8 | 공통2 참조 |
| 매출액 | 계약금액 컬럼 | 계약서 총액 |
| 비목별 금액 | 집행계획 컬럼 | 산출내역서 합계 참조 |
| 영업이익 A | 자동계산 | 매출액 - 합계 |
| 특기사항 | D열 | 계약서 내용 |

## Reviewer 금액 검증 체계

검증 규칙 상세: `.claude/agents/reviewer.md` 참조.

요약:
- 행별 산술: 수량 × 단가 = 금액 (1원 오차 FAIL)
- 시트 간 교차: 5.4 수수료 ↔ 5. 산출내역서 ↔ 0. 갑지 일치
- 소스 대조: 계약서 총액 = 갑지 매출액 / 협력사 견적서 = 수수료 집행금액
- 마진 구조: 집행단가 ≤ 계약단가 (역전 시 FAIL)
- 5단계 검증: 수수료 구조 → 충돌 해결 → 산출내역서 → 갑지 → 기본 정보

## [추측] 알람 발생 가능 항목

프로젝트마다 달라지는 항목 중 문서에 명시되지 않을 수 있는 항목:
- 수수료 코드 분류 (용역/유지보수/기타 — 견적서 성격으로 판단 필요)
- 보험료 요율 (매년 변경, 공지 필요)
- 상여금 지급 시점/대상 (계약서에 없을 경우)
- 발주처 vs 계약처 구분 (문서에 명확히 구분 안 된 경우)
- 협력사 견적서 단가의 계약 컬럼 매핑 (계약서에 협력사 내역 없는 경우)
